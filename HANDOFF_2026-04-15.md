  - **Semantic Model Execute Queries REST API** (the renamed "Dataset Execute Queries REST API")
  - **Service principals can call Fabric public APIs**
  - **Allow XMLA endpoints and Analyze in Excel with on-premises semantic models**
  - **Users can edit semantic models in the Power BI service**
- SP `PowerBIEmbedApp` membership on workspace "Charlie": **Admin** with "All permissions granted"

- `impersonatedUserName: "charles@haleglobal.com"` sent in the executeQueries body

### Definitive test that isolated the cause

Microsoft's executeQueries "Try It" page, signed in as `charles@haleglobal.com` (user auth, not SP):

- Same workspace id, same dataset id, same query (`EVALUATE ROW("ping", 1)`)
- Returned **200** with `{"results":[{"tables":[{"rows":[{"[ping]":1}]}]}]}`

**Conclusion:** The dataset is fine. The workspace is fine. The tenant settings are fine. The API call is syntactically valid. Power BI specifically rejects service principals for `executeQueries` against this dataset even when the SP has workspace Admin + dataset Admin.

This appears to be an undocumented platform-level block that we can't resolve from the outside. We stopped debugging SP and decided to pivot.

## Plan for tomorrow: user-delegated auth (Path 3)

Instead of authenticating as a service principal, the pipeline will authenticate as `charles@haleglobal.com` using a refresh token. This is proven to work with executeQueries.

### Steps needed (~30 min of my work + ~5 min of yours)

1. **Enable public client flow on PowerBIEmbedApp.** Azure Portal → App Registrations → PowerBIEmbedApp → Authentication → toggle "Allow public client flows" to **Yes** → Save.

2. **I build `scripts/bootstrap_user_auth.py`** — local interactive script using MSAL's device code flow. You run it once: it prints a URL and a code, you visit the URL and enter the code, sign in as `charles@haleglobal.com`, approve. Script outputs a refresh token.

3. **You add the refresh token as a Render env var** `MSAL_REFRESH_TOKEN`.

4. **I update `src/pipeline/powerbi.py`** to check for `MSAL_REFRESH_TOKEN` and use MSAL's `acquire_token_by_refresh_token` instead of the SP client-credentials flow when present. Falls back to SP if the env var is absent (so we can flip back if SP ever gets fixed by Microsoft).

5. **Every 60-90 days you'll need to re-bootstrap** — run the local script, paste new refresh token into Render env. 1-minute quarterly maintenance task.

### Open question to decide tomorrow

Where does the rotating refresh token live long-term?

- **Option A (simplest):** Just env var on Render. Bootstrap produces new token every quarter, you paste in. Works fine.
- **Option B (more robust):** Render persistent disk ($1/mo for 1 GB). MSAL persists its SerializableTokenCache to disk and silently refreshes. No manual rotation. But: Render Starter may not support persistent disks — would need to upgrade to Standard.
- **Option C (overkill):** Secret manager like AWS Secrets Manager or Azure Key Vault. Extra infra, extra cost.

Recommend starting with A, upgrading to B later if the rotation becomes annoying.
